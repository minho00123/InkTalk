# 💡 동기

저희 팀은 자바 소켓을 주제로 미니 프로젝트를 진행하면서, "어떻게 하면 더 효율적으로 소통할 수 있을까?"라는 질문에서 출발했습니다. 단순한 텍스트 기반의 채팅만으로는 실시간 협업 과정에서 모든 아이디어를 효과적으로 전달하기 어렵다고 느꼈고, 특히 시각적인 정보나 복잡한 개념을 설명할 때 한계를 경험했습니다.

이에 저희는 실시간으로 그림을 그리며 아이디어를 공유할 수 있는 화이트보드 기능을 추가하면, 더 직관적이고 몰입감 있는 커뮤니케이션이 가능하리라는 판단을 내렸습니다. 이러한 문제의식과 아이디어를 바탕으로, 저희는 여러 사용자가 동시에 접속하여 자유롭게 그림을 그리고 대화할 수 있는 협업용 화이트보드 회의 도구를 개발하게 되었습니다.

이 프로젝트는 단순히 기술 구현을 넘어서, 실제 협업 환경에서의 불편함을 해소하고자 하는 고민에서 시작되었으며, 사용자 중심의 실용적인 도구를 만들고자 하는 목표를 가지고 진행되었습니다.

# 🔥 도전

프로젝트를 진행하며 가장 큰 난관은 한 명의 사용자가 접속하면 다른 사용자가 접속할 수 없는 문제였습니다.

처음에는 화이트보드 기능과 채팅 기능을 각각 독립적으로 개발했기 때문에, 두 기능이 각각 별도의 Socket 객체를 생성하여 서버에 연결하고 있었습니다. 이로 인해 한 명의 클라이언트가 두 번 접속하는 형태가 되었고, 서버는 이를 서로 다른 클라이언트로 인식해 서버 자원을 중복으로 할당하는 문제가 발생했습니다.

특히 저희 서버는 클라이언트당 하나의 연결만 허용하도록 설계돼 있었기 때문에, 새로운 사용자가 접속을 시도할 경우 기존 연결이 끊기거나, 접속이 아예 차단되는 심각한 문제가 생겼습니다.

이 문제를 해결하기 위해 저는 다음과 같은 구조 개선을 진행했습니다:
- 화이트보드와 채팅 기능을 하나의 Board 클래스로 통합
- 단일한 Socket 객체만 생성하여, 모든 데이터(그림, 채팅 메시지 등)를 하나의 연결로 처리할 수 있도록 설계 변경
- 통신 데이터에 타입 정보를 포함시켜, 서버에서 수신된 데이터가 화이트보드 관련인지 채팅 메시지인지 구분할 수 있도록 처리

이러한 구조 개편을 통해 문제를 해결할 수 있었으며, 동시에 다음과 같은 점을 배울 수 있었습니다:
- 소켓 통신 구조에 대한 실질적인 이해가 깊어졌고,
- 팀 프로젝트에서 기능을 나누어 개발할 때, 각 기능의 통신 구조가 어떻게 연결되는지에 대한 사전 논의와 소통이 얼마나 중요한지 깨달았습니다.

# 📝 회고
이번 프로젝트는 Java를 활용해 처음으로 직접 구현해 본 프로그램이었기에 부족한 부분도 많았지만, 그만큼 배운 것도 많았습니다. 특히 소켓 통신에 대한 이해가 깊어졌고, 본격적인 웹 개발로 넘어가기 전에 네트워크 통신의 기본 원리를 체험해볼 수 있는 소중한 기회였습니다.

프로젝트 초기에 기획했던 모든 기능을 구현하지는 못했습니다. 예를 들어, 사용자가 여러 개의 방을 생성하고 입장할 수 있는 구조를 도입하고자 했지만, 제한된 시간 안에 프로젝트를 완성해야 했기 때문에 일부 기능은 과감히 포기할 수밖에 없었습니다.

또한 이번 경험을 통해 테스트의 중요성을 절실히 느꼈습니다. 예상치 못한 다양한 에러가 발생했고, 그때마다 충분한 시간을 들여 다양한 상황을 테스트하고 문제를 해결하는 과정이 얼마나 중요한지 깨달았습니다.

결과적으로 이번 프로젝트를 통해 얻은 가장 큰 교훈은, 기능을 많이 넣는 것보다 안정적이고 완성도 높은 프로그램을 만드는 것이 더욱 가치 있다는 점이었습니다. 다음 프로젝트에서는 기획과 구현, 테스트를 더 균형 있게 구성하여, 완성도 높은 결과물을 만드는 데 집중하고 싶습니다.
